```
  body: { sport: string, year: number, setName: string }

       (empty array when nothing scored >= 0.7)
  200: { matches: [], reason: "tcdb-unavailable" }  // Cloudflare block

  200: { metadata: null, reason: "tcdb-unavailable" }
  400: { error: "Invalid request body: ..." }
  500: { error: "TCDB get-set failed" }
```

**Deviations from spec:**

1. **The audit doc's "BSC + SL captured-but-dropped" claim was partially inaccurate.** Reading the code: `commitCardChecklist` already persists `attributes`, `printRun`, `autographType`, `cardVariation`, `isRookie`, `isRelic` end-to-end (typed columns on `cardChecklist`). What was MISSING was the surface of those values as **feature-map keys** for marketplace-agnostic consumption. Stage 3b lands that surface (derivation into `cardChecklist.features`) rather than rewiring data that was already flowing. BSC's bulk-upload endpoint genuinely doesn't carry `printRun` or `autographType` (adapter sets both to `undefined`), so there's nothing to harvest there — the audit overstated availability. SL's tokenizer-derived values do flow through correctly.

2. **`existing` cardChecklist rows are NOT patched with derived features on re-commit.** This matches the Stage 2 propagation contract (`setSelectorOptionFeature` is the single source of truth for mutating existing card features). A re-fetch of an existing set won't backfill features onto pre-Stage-3b rows — those need an admin to trigger `setSelectorOptionFeature` (Stage 4 SetFeaturesPanel) or be deleted + re-inserted.

3. **`storeSelectorOptions` / `storeReconciledOptions` / `addCustomSelectorOption` were NOT given scheduler hooks.** Those mutations create rows but don't write card content — TCDB enrichment is much more useful AFTER the set has cards. Hooking enrichment into `commitCardChecklist` (the only path that creates cards) gives a single, predictable trigger point. If admins want to enrich a set without committing cards (e.g. just metadata for a freshly reconciled row), a follow-up could add an explicit "Refresh metadata" button that schedules the same internalAction.
