
**Key design decisions:**

1. **Double-write pattern for idempotency.** SetFeaturesPanel smoke and features-propagation
   both write a sentinel value first, then the intended value. This guarantees the mutation
   fires even if a previous run left the same intended value in the DB.

2. **`requires:cards-loaded` for all new flows.** These flows all work against the
   Baseball / 2024 / Topps Chrome / Base subtree which is owned by the cascade. The dep
   graph ensures the cascade has populated the hierarchy before these flows run.

3. **Unique card numbers (777, 888, 889, 991, 992).** Each flow uses a distinct card
   number and deletes its card at the end. The 700-999 range is reserved for test
   management cards — unlikely to conflict with marketplace card numbers (which typically
   start at 1).

4. **PropTest- sentinel prefix.** The features-propagation flow uses "PropTest-MLB",
   "PropTest-NL", "PropTest-AL" as League values to distinguish its writes from the
   smoke test's "MLB" value. This prevents false idempotency where the old value equals
   the new value and the propagation engine returns a no-op.

5. **`id:` not `text:` for aria-label buttons.** Followed the project rule: aria-label
   maps to resource-id in Maestro web. `tapOn: id: "Add team"` works; `tapOn: "Add team"`
   does not (plain string matches visible text, not aria-label).
