## The rules (shorthand) — ⭐ REFINED 2026-06-15 (R4/R5 tightened + R9) · R10 added 2026-06-18
R1 what feature · R2 assert that & only that (no silent `when:` fall-through) · R3 dedup (reuse shared logic; + no other flow tests the same thing — NOT intra-flow before/after repetition) · R4 **flows NEVER log in** · R5 **everything reacts in 7s** · R6 no redundant `assertVisible` after `scrollUntilVisible` · R7 no destructive data actions · R8 `centerElement: true` on scroll-to-tap · R9 **all drilling via the drill utils** · R10 **ensure no extra waits** (no guard/branch that silently burns a full timeout in the warm path).

### ⭐ Refined criteria (2026-06-15) — the load-bearing tightening
- **R4 — no IN-BODY re-login / credential-setup in flows (KEEP the `/testing/sign-in` entry).** Every
  authenticated flow ENTERS via `url: …/testing/sign-in?redirect=<route>&worker=<N>` — this is the **required
  per-flow Clerk auth handshake**, not a removable redirect. Proven (2026-06-15): `launchApp` starts an
  **unauthenticated** browser state, `/set-selector` is auth-gated, so a flow pointed straight at `/set-selector`
  **bounces to the Clerk `/signin` page** (screenshot from #35's failed direct-nav). `/testing/sign-in` runs a real
  Clerk ticket sign-in each run (warm ~5.6s; under parallelism it can hit Clerk rate-limit **retries** → keep the
  `45000` sign-in wait, matching every green flow). **What the audit removed (and stays banned):** the redundant
  **in-body** credential-setup / re-sign-in STEPS (`runFlow setup-bsc/sl-credentials`, explicit re-auth) — flows
  rely on the bootstrap-seeded creds. All **33** authenticated audited flows keep the `/testing/sign-in` entry;
  only the public `home/*` pages and the `util-*` sub-flows (run inside a parent's session) enter directly.
  - ⚠️ Two earlier notes in this doc — "fresh browser per flow / harness change needed" AND "session persists /
    hop is removable" — were BOTH WRONG (the persistent chromedriver *process* ≠ a persistent auth *session*;
    `launchApp` drops the session, `/testing/sign-in` re-auths). #35/#36 had the hop wrongly removed → restored.
  - Open (optional, separate): if a truly carried-over per-worker session is wanted (zero per-flow auth), that's a
    **harness investigation** — why `launchApp` doesn't inherit the bootstrap session.
- **R5 — everything reacts in 7s.** Every wait is the **7s default**. The ONLY long-wait exception is a step
  that directly drives a **LIVE marketplace round-trip** (a real BSC/SportLots fetch or auth) **and only when
  the data is not already pre-synced**. EVERYTHING else — page loads, column renders, and **ALL Convex backend
  calls** (entity-create, card/metadata save, **feature propagation to N cards**, sync-status reads) — must
  react in 7s. A `>7s` non-marketplace response is a **FINDING to fix in the product** (optimize / make async /
  optimistic), **never a wait to inflate.** Only the **setup track** keeps cold-sync long waits (it is the one
  place that deliberately warms the marketplaces). _The old "bulk-write carve-out" (30s propagation, 30s
  cold-commit) is RETIRED — those are 7s now; if they exceed it, that's the finding._
- **R9 — all set-builder drilling goes through the drill utils** (`util-drill-to-2024-topps-chrome` /
  `util-drill-to-custom`). No flow hand-rolls its own Sport→Year→Manufacturer→Set→Variant drill. The util is
  the single source of the drill logic + its (marketplace-exception) waits. Hand-rolled inline drills are a
  violation → replace with `runFlow` the util.
- **R10 — ensure no extra waits (added 2026-06-18, from the drill profiling).** Beyond R5's "no inflated
  timeouts," hunt for waits that burn time while doing NO real synchronization. The canonical case: a
  `when: { notVisible: "<X>" }` guard where X is normally PRESENT — Maestro polls the FULL default (~7s)
  before resolving false, every single time. The real-path drill had ~6 such guards (~42s) PLUS an
  "Add custom Sets" create-detour for an already-existing set (~22s); deleting both cut
  `util-drill-to-2024-topps-chrome` from ~100s → ~32s (search-select only, mirroring `setup.yaml`). When
  walking a flow, audit for: `notVisible:` guards on normally-present elements, create/fallback branches
  that run when the entity already exists, and redundant `extendedWaitUntil`/settle after an action.
  ⚠️ NOTE: the residual ~2s/Maestro-web tap is `hierarchyBasedTap`'s view-hierarchy-change wait — it is NOT
  `waitToSettleTimeoutMs` and NOT tunable from flow YAML (proven by experiment, two green runs unchanged).
  Don't add settle waits expecting a speedup; the only levers below ~32s are fewer actions or a deep-link
  route (app change). See agent-memory `reference_maestro_web_tap_cost`.
